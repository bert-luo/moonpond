# Analyze Game

Perform a thorough post-mortem analysis of a game generated by the moonpond
contract pipeline. The goal is to identify every failure, trace each one back to
the specific pipeline stage code that caused it, and write up findings with
code snippets so the user can understand and fix the root causes.

## Input

The user provides a path to a generated game directory, e.g.:
`/Users/albertluo/other/moonpond/games/<game-name>_<timestamp>`

## Analysis Flow

### 1. Inspect the Output

- **Read `output.png`** — the screenshot of the running game in the browser.
  Note what is visible vs. what is missing or broken.
- **Read `intermediate/5_result.json`** — check if export succeeded and what
  controls were registered.

### 2. Read All Generated Code

Read every file in `project/` (excluding `.godot/` internal cache):
- All `.gd` scripts
- All `.tscn` scene files
- `project.godot` configuration

For each file, note:
- What node type it extends
- What `@onready` references it expects (e.g., `%NodeName`)
- What signals it connects to
- What scenes it preloads
- What groups it uses

### 3. Read Intermediate Pipeline Artifacts

- `intermediate/1_rich_game_spec.json` — the expanded game spec
- `intermediate/2_game_contract.json` — the node contracts (interfaces)
- `intermediate/2.5_game_manager.gd` — the generated game manager
- `intermediate/3_node_files/` — per-node generated scripts
- `intermediate/4_wiring_files/` — Main.tscn and project.godot from wiring stage

### 4. Cross-Reference Against Pipeline Source Code

For each issue found, trace it back to the pipeline stage that caused it.
Read the relevant pipeline source files:

- `backend/backend/stages/node_generator.py` — node generation prompts and logic
- `backend/backend/stages/wiring_generator.py` — scene tree assembly prompts and logic
- `backend/backend/stages/contract_generator.py` — contract generation
- `backend/backend/stages/game_manager_generator.py` — game manager generation
- `backend/backend/stages/spec_expander.py` — spec expansion
- `backend/backend/stages/exporter.py` — project assembly and WASM export
- `backend/backend/pipelines/contract/pipeline.py` — pipeline orchestration

### 5. Identify Failures

For each failure, document:

1. **What's broken** — observable symptom (from screenshot or code inspection)
2. **Why it's broken** — the technical cause in the generated code
3. **Pipeline root cause** — which pipeline stage and what specific code
   (LLM prompt text, Python logic, or missing logic) caused the issue
4. **Code snippets** — include relevant excerpts from:
   - The generated game code showing the bug
   - The pipeline Python code or LLM prompt that produced it
   - The contract/spec that defined the expectation

### 6. Categorize by Severity

- **P0 — Game-breaking**: Game cannot start, crashes on load, core mechanic non-functional
- **P1 — Major**: Significant feature missing, gameplay severely degraded
- **P2 — Minor**: Visual glitch, non-critical logic error, code smell

### 6.5. Classify Fix Type

For each failure, determine whether it is:

- **One-off**: A fluke LLM hallucination or game-specific mistake that wouldn't
  reliably recur across different games. No pipeline change needed.
- **Systemic**: A gap in the pipeline (missing prompt guidance, flawed Python logic,
  missing generation step) that would likely recur for other games with similar
  features. Warrants a general pipeline fix.

For systemic issues, propose a concrete fix: which file to change, what to add or
modify, and why it would prevent the class of failure — not just this instance.

### 7. Write Report

Write findings to `report.md` in the game directory (sibling to `output.png`,
`project/`, `intermediate/`, `export/`).

Use this structure:

```markdown
# Game Analysis Report: <Game Title>

**Generated**: <timestamp from directory name>
**Pipeline**: contract
**Status**: <BROKEN | DEGRADED | FUNCTIONAL>

## Screenshot Observations

<What the output.png shows — what works, what's visibly wrong>

## Failures

### P0: <Issue Title>

**Symptom**: <What's broken>

**Generated code** (`<file>`):
\```gdscript
<relevant snippet>
\```

**Pipeline cause** — `<stage file>`:
<Explanation of what the pipeline stage did wrong>

\```python
<relevant pipeline code snippet>
\```

**Contract expectation** (`2_game_contract.json`):
\```json
<relevant contract snippet>
\```

**Fix type**: <One-off | Systemic>
<If systemic: one-line summary of the general pipeline fix needed>

---

<repeat for each failure>

## Summary

| Priority | Count | Issues |
|----------|-------|--------|
| P0 | N | ... |
| P1 | N | ... |
| P2 | N | ... |

### Root Cause Themes

<Group failures by pipeline-level root cause — e.g., "node generator doesn't
produce .tscn files", "wiring generator doesn't create child nodes">

### Pipeline Fix Candidates

List only the **systemic** issues that warrant general pipeline changes.
Do not include one-off LLM hallucinations here.

For each:
1. **Issue class** — the general pattern, not the game-specific instance
2. **Affected stage** — which pipeline file(s)
3. **Proposed fix** — concrete change (prompt addition, Python logic change, new step)
4. **Expected impact** — what class of future games this would help
```

## Guidelines

- Always include code snippets — both from the generated game AND from the
  pipeline source that caused the issue. This is critical for understanding.
- Be specific about file paths and line numbers.
- Don't speculate about issues — verify by reading the actual code.
- Check for missing files that are referenced (preload, ExtResource, etc.).
- Check signal connections in .tscn against actual signal definitions in scripts.
- Check @onready references against actual scene tree structure.
- Check for initialization issues (null defaults, missing _ready setup).
- Note any hardcoded values that should be dynamic (screen sizes, etc.).
