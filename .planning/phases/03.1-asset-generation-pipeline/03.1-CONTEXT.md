# Phase 3.1: Asset Generation Pipeline - Context

**Gathered:** 2026-03-14
**Status:** Ready for planning
**Source:** User direction + PRD.md + Phase 3 context

<domain>
## Phase Boundary

This phase adds AI image generation to the multi-stage pipeline built in Phase 3. It introduces a configurable `ImageProvider` protocol so the user can swap between OpenAI (DALL-E) and Google (Imagen) without code changes. A new Asset Generator stage produces sprites, backgrounds, and textures that get written into the Godot project before export.

Deliverable: The existing multi_stage pipeline gains an Asset Generator stage that produces `.png` files in `assets/generated/`, configurable via `IMAGE_PROVIDER` env var, with `.env` loading via python-dotenv.

</domain>

<decisions>
## Implementation Decisions

### Provider Architecture
- `ImageProvider` Protocol in `backend/backend/providers/base.py` with a `generate_image(prompt, size, style) -> bytes` async method
- OpenAI provider (`providers/openai.py`) wraps DALL-E 3 API
- Google provider (`providers/google.py`) wraps Imagen API
- Provider registry resolves from `IMAGE_PROVIDER` env var (default: `openai`)
- Providers are instantiated once and shared across pipeline runs

### Environment / .env Loading
- `python-dotenv` added as dependency; loaded at app startup in `main.py`
- `.env` at repo root contains: ANTHROPIC_API_KEY, OPENAI_API_KEY, GOOGLE_API_KEY, TRIPO_API_KEY
- `.env` is gitignored
- `.env.example` committed with placeholder keys for onboarding
- Phase 3's Anthropic stages should also benefit from dotenv loading (retroactive improvement)

### Asset Generator Stage
- Positioned between Game Designer and Code Generator in the pipeline sequence
- Takes `GameDesign` as input — uses title, genre, visual_style, scenes to determine what assets to generate
- Outputs an `AssetManifest` (dict of asset_name → file_path) that Code Generator and Visual Polisher can reference
- Generates: character sprites, background images, key object textures (3-6 images per game, budget-conscious)
- Each image saved as `.png` in `games/{job_id}/project/assets/generated/`
- Emits `ProgressEvent(type='stage_start', message='Generating visual assets...')` at start

### Pipeline Integration
- `MultiStagePipeline` updated: Prompt Enhancer → Game Designer → **Asset Generator** → Code Generator → Visual Polisher → Exporter
- Code Generator's system prompt updated to reference generated assets when available
- Visual Polisher aware of generated assets — can reference them in shader/material assignments
- Asset generation is best-effort: if image gen fails, pipeline continues with template assets only (graceful degradation)

### Image Generation Strategy
- Game Designer's `VisualStyle` output guides image prompts (art style, color palette, mood)
- Asset Generator constructs image prompts from game context — not raw user prompt
- Target image sizes: sprites 256x256, backgrounds 1024x576, textures 512x512
- DALL-E 3: use "natural" style for game art; Google Imagen: equivalent quality setting

### Claude's Discretion
- Exact prompt engineering for image generation (how to translate GameDesign → image prompts)
- Whether to generate images in parallel or sequentially
- How many images per game (balance between quality and cost/latency)
- Caching strategy for generated images (if any)
- Error handling per-image vs per-batch
- How Code Generator references generated assets in GDScript (load() paths)

</decisions>

## Specific Ideas

### PRD References
- PRD line 74: `OpenAI (secondary/image)` in tech stack
- PRD line 283: `OpenAI image generation for sprites, Tripo for 3D assets` in ROMA description
- Core value: "visual and aesthetic quality" as primary differentiator

### Godot Asset Loading Pattern
```gdscript
# How generated assets are loaded in GDScript
var sprite_texture = load("res://assets/generated/player_sprite.png")
$Sprite2D.texture = sprite_texture

var bg_texture = load("res://assets/generated/background.png")
$Background.texture = bg_texture
```

### Provider Interface Shape
```python
class ImageProvider(Protocol):
    async def generate_image(
        self,
        prompt: str,
        width: int,
        height: int,
        style: str | None = None,
    ) -> bytes: ...
```

### Asset Types to Generate
| Type | Purpose | Size | Count per game |
|------|---------|------|----------------|
| Character sprite | Player, enemies, NPCs | 256x256 | 1-3 |
| Background | Scene backgrounds | 1024x576 | 1-2 |
| Texture | Ground, walls, objects | 512x512 | 1-2 |
