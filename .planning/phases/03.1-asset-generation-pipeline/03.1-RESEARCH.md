# Phase 3.1: Asset Generation Pipeline - Research

**Researched:** 2026-03-14
**Domain:** OpenAI image generation SDK, Google Imagen SDK, python-dotenv, Godot 4 asset import/export, async Python strategy pattern
**Confidence:** HIGH (core SDKs verified; Godot import behavior confirmed from official issue tracker)

## Summary

Phase 3.1 adds AI image generation to the multi-stage pipeline. An `ImageProvider` protocol with OpenAI and Google implementations allows swapping providers via `IMAGE_PROVIDER` env var. An Asset Generator stage produces sprite, background, and texture `.png` files that Godot loads during headless export.

The most critical non-obvious finding is about **Godot's import system**: PNG files placed in `res://` must be processed by Godot's import pipeline to produce `.ctex` files before export. The headless `--export-release` command implies `--import` as of Godot 4.2.2, so writing PNGs into the project directory before calling `run_headless_export()` is sufficient — Godot will import them automatically on the export run. This means the Asset Generator's `.png` files written to `games/{job_id}/project/assets/generated/` will be picked up correctly as long as they are present before the Exporter stage runs.

For OpenAI, **DALL-E 3 is deprecated** (end-of-life 05/12/2026) — the current recommended model is `gpt-image-1`, which returns base64-encoded bytes by default, uses `AsyncOpenAI` for async operation, and supports `1024x1024`, `1536x1024`, and `1024x1536` sizes. For Google, the `google-genai` SDK uses `client.aio.models.generate_images()` for async operation with `imagen-4.0-generate-001` as the current stable model; image bytes are accessed via `generated_image.image.image_bytes`.

**Primary recommendation:** Implement `ImageProvider` Protocol, `OpenAIImageProvider` using `gpt-image-1` via `AsyncOpenAI`, `GoogleImageProvider` using Imagen 4 via `google-genai`, and an `AssetGeneratorStage` that writes PNG files before the Exporter runs.

### Pattern 1: ImageProvider Protocol
**What:** A `typing.Protocol` defining the async image generation interface. Providers implement it; the registry resolves from env var.
**When to use:** Any time image generation is needed — the caller never imports a concrete provider.

```python
# backend/backend/providers/base.py
from __future__ import annotations
from typing import Protocol


class ImageProvider(Protocol):
    async def generate_image(
        self,
        prompt: str,
        width: int,
        height: int,
        style: str | None = None,
    ) -> bytes:
        """Generate an image and return raw PNG bytes."""
        ...
```

### Pattern 2: OpenAI Provider (gpt-image-1)
**What:** `AsyncOpenAI.images.generate()` with `gpt-image-1` model; response returns `b64_json` which is decoded to bytes.
**When to use:** `IMAGE_PROVIDER=openai` (default).

Note on sizes: gpt-image-1 supports `1024x1024`, `1536x1024`, `1024x1536`. The CONTEXT.md specifies 256x256 sprites and 512x512 textures — these sizes are NOT supported by gpt-image-1. The provider must map requested sizes to the nearest supported size (`1024x1024` for all cases) and let the GDScript scale the texture via `Sprite2D.scale` or similar.

```python
# Source: https://platform.openai.com/docs/guides/image-generation
# backend/backend/providers/openai.py
from __future__ import annotations
import base64
import os
from openai import AsyncOpenAI


class OpenAIImageProvider:
    def __init__(self) -> None:
        self._client = AsyncOpenAI()  # reads OPENAI_API_KEY from env

    async def generate_image(
        self,
        prompt: str,
        width: int,
        height: int,
        style: str | None = None,
    ) -> bytes:
        # gpt-image-1 only supports 1024x1024, 1536x1024, 1024x1536
        size = _map_size(width, height)
        result = await self._client.images.generate(
            model="gpt-image-1",
            prompt=prompt,
            size=size,
            n=1,
        )
        # gpt-image-1 always returns b64_json
        return base64.b64decode(result.data[0].b64_json)


def _map_size(width: int, height: int) -> str:
    """Map requested dimensions to nearest gpt-image-1 supported size."""
    if width > height:
        return "1536x1024"
    elif height > width:
        return "1024x1536"
    return "1024x1024"
```

### Pattern 3: Google Provider (Imagen 4)
**What:** `client.aio.models.generate_images()` with `imagen-4.0-generate-001`; image bytes accessed via `.image.image_bytes`.
**When to use:** `IMAGE_PROVIDER=google`.

Note on auth: Gemini Developer API uses `GOOGLE_API_KEY` (or `GEMINI_API_KEY`). The `genai.Client()` reads `GOOGLE_API_KEY` automatically from env. The CONTEXT.md specifies `GOOGLE_API_KEY` — this matches.

Imagen 4 does not support arbitrary pixel dimensions — it uses named sizes (`"1K"` default, `"2K"` premium). All requests should use `"1K"` for cost efficiency.

```python
# Source: https://ai.google.dev/gemini-api/docs/imagen
# backend/backend/providers/google.py
from __future__ import annotations
from google import genai
from google.genai import types


class GoogleImageProvider:
    def __init__(self) -> None:
        self._client = genai.Client()  # reads GOOGLE_API_KEY from env

    async def generate_image(
        self,
        prompt: str,
        width: int,
        height: int,
        style: str | None = None,
    ) -> bytes:
        response = await self._client.aio.models.generate_images(
            model="imagen-4.0-generate-001",
            prompt=prompt,
            config=types.GenerateImagesConfig(
                number_of_images=1,
                output_mime_type="image/png",
            ),
        )
        return response.generated_images[0].image.image_bytes
```

### Pattern 4: Provider Registry
**What:** A dict mapping env var values to provider instances; resolved once at startup.
**When to use:** This is the ASSET-05 mechanism — no code changes needed to swap providers.

```python
# backend/backend/providers/base.py (continued)
import os

_REGISTRY: dict[str, type] = {}

def get_image_provider() -> ImageProvider:
    """Resolve provider from IMAGE_PROVIDER env var. Default: openai."""
    provider_name = os.getenv("IMAGE_PROVIDER", "openai").lower()
    from .openai import OpenAIImageProvider
    from .google import GoogleImageProvider
    _REGISTRY.update({"openai": OpenAIImageProvider, "google": GoogleImageProvider})
    cls = _REGISTRY.get(provider_name)
    if cls is None:
        raise ValueError(f"Unknown IMAGE_PROVIDER '{provider_name}'. Use: openai, google")
    return cls()
```

### Pattern 5: Asset Generator Stage
**What:** Takes `GameDesign`, constructs image prompts from game context, generates images in parallel, writes PNG bytes to disk, returns `AssetManifest`.
**When to use:** Between Game Designer and Code Generator in the pipeline.

```python
# backend/backend/stages/asset_generator.py
from __future__ import annotations
import asyncio
from pathlib import Path
from ..providers.base import ImageProvider
from ..pipelines.base import EmitFn, ProgressEvent
from .models import GameDesign, AssetManifest

GAMES_DIR = Path(__file__).parent.parent.parent.parent.parent / "games"


async def run_asset_generator(
    provider: ImageProvider,
    game_design: GameDesign,
    job_id: str,
    emit: EmitFn,
) -> AssetManifest:
    await emit(ProgressEvent(type="stage_start", message="Generating visual assets..."))

    assets_dir = GAMES_DIR / job_id / "project" / "assets" / "generated"
    assets_dir.mkdir(parents=True, exist_ok=True)

    # Build asset spec list from game design
    asset_specs = _build_asset_specs(game_design)

    # Generate in parallel (best-effort: failures yield None)
    results = await asyncio.gather(
        *[_generate_one(provider, spec, assets_dir) for spec in asset_specs],
        return_exceptions=True,
    )

    # Build manifest — only include successful generations
    manifest: AssetManifest = {}
    for spec, result in zip(asset_specs, results):
        if isinstance(result, Exception):
            # Graceful degradation: log, continue without this asset
            continue
        manifest[spec["name"]] = f"res://assets/generated/{spec['filename']}"

    return manifest
```

### Pattern 7: GDScript Asset Reference
**What:** Code Generator uses `load("res://assets/generated/...")` for generated assets; falls back to no texture if asset not present.
**When to use:** System prompt for Code Generator must include the AssetManifest.

```gdscript
# How generated assets are loaded in GDScript (from CONTEXT.md)
var sprite_texture = load("res://assets/generated/player_sprite.png")
$Sprite2D.texture = sprite_texture

# Defensive pattern if asset may not exist:
var bg_path = "res://assets/generated/background.png"
if ResourceLoader.exists(bg_path):
    $Background.texture = load(bg_path)
```

### Anti-Patterns to Avoid
- **Using DALL-E 3 instead of gpt-image-1:** DALL-E 3 is deprecated, end-of-life May 2026. Use `gpt-image-1`.
- **Requesting non-standard sizes from image APIs:** gpt-image-1 only supports `1024x1024`, `1536x1024`, `1024x1536`. Imagen 4 uses named sizes `"1K"` / `"2K"`. Map all size requests to supported values.
- **Creating `AsyncOpenAI` at module level:** Same rule as `AsyncAnthropic` — create in `__init__()`, not at module scope.
- **Writing PNG files AFTER the Exporter runs:** Assets must be written to the project directory BEFORE `run_headless_export()` is called so Godot can import them.
- **Using `google-generativeai` (old SDK):** The legacy library is deprecated. Use `google-genai` (`from google import genai`).
- **Calling `load_dotenv()` after FastAPI app creation:** Must be called before any code that reads env vars, especially before pipeline class instantiation.
- **Hard-coding asset paths in GDScript:** Code Generator must use `res://assets/generated/{name}` paths from the AssetManifest, not guessed names.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| HTTP connection pooling for image APIs | Per-call `AsyncOpenAI()` / `genai.Client()` | Single shared instance per provider | SDKs manage connection pools; per-call creates new connections |
| Rate limit retry for image gen | Custom backoff loop | OpenAI SDK auto-retries; add `max_retries` param | SDK handles 429/529 with exponential backoff |
| Base64 → bytes decoding | Custom decoder | `base64.b64decode(result.data[0].b64_json)` | One-liner stdlib |
| PNG file write | Custom binary writer | `Path.write_bytes(image_bytes)` | One-liner pathlib |
| Env var config system | Custom config class | `load_dotenv()` + `os.getenv()` | Sufficient for this use case; pydantic-settings would add complexity |

**Key insight:** Both image SDKs abstract all HTTP, auth, and serialization. The provider implementation is thin: call SDK, extract bytes, return.

## Common Pitfalls

### Pitfall 1: DALL-E 3 Unsupported Sizes Cause API Error
**What goes wrong:** Passing `size="256x256"` to `dall-e-3` (or `gpt-image-1`) raises `InvalidRequestError: size must be one of...`
**Why it happens:** DALL-E 3 and gpt-image-1 do not support small sizes; DALL-E 2 supported 256x256 and 512x512
**How to avoid:** Always map requested dimensions to a supported size before calling the API. For gpt-image-1: use `1024x1024` for square, `1536x1024` for landscape, `1024x1536` for portrait. Scale in GDScript.
**Warning signs:** `openai.BadRequestError: size` in pipeline logs

### Pitfall 2: Google SDK Requires Billing Account
**What goes wrong:** Imagen API call raises `PermissionDenied: Imagen API is only accessible to billed users`
**Why it happens:** Imagen (unlike Gemini text) requires a billing-enabled project even for low usage
**How to avoid:** Use a billing-enabled Google Cloud project. The `.env.example` should note this requirement. Test with OpenAI provider first; only test Google provider with a verified billing account.
**Warning signs:** `google.api_core.exceptions.PermissionDenied` at `generate_images()` call

### Pitfall 3: PNG Files Written After Export Starts
**What goes wrong:** Generated PNGs are missing from the exported WASM; game loads without textures
**Why it happens:** Pipeline stage order error — Exporter called before Asset Generator finishes
**How to avoid:** Asset Generator must be `await`ed before Exporter stage begins. The `assets/generated/` directory must exist and contain PNG files before `run_headless_export()` is called.
**Warning signs:** Export succeeds but game shows blank/missing textures at runtime

### Pitfall 4: Godot Import Step for Newly Added PNGs
**What goes wrong:** Headless export log shows import errors for PNG files added to project directory
**Why it happens:** Older understanding that headless export doesn't import. In Godot 4.2.2+, `--export-release` implies `--import` — this is handled automatically.
**How to avoid:** No action needed if Godot >= 4.2.2. The existing `run_headless_export()` command is sufficient. Do not add a separate `--import` step.
**Warning signs:** Only if Godot < 4.2.2 (check `godot --version`). Confirmed that project uses Godot 4.5.1 (SETUP-01), so this is handled.

### Pitfall 5: Parallel Generation Blows API Rate Limits
**What goes wrong:** `asyncio.gather()` with 6 concurrent image generation requests triggers OpenAI rate limit (429)
**Why it happens:** Default OpenAI tier allows limited concurrent requests; firing 6 at once can exceed it
**How to avoid:** The OpenAI SDK auto-retries with backoff. However, for > 4 images, consider sequential batches or a semaphore (`asyncio.Semaphore(3)`) to limit concurrency. Start with full parallel (3-6 images typical budget), monitor for rate limit errors.
**Warning signs:** `RateLimitError` or `openai.APIStatusError: 429` in pipeline logs during asset generation

### Pitfall 6: `generate_image()` Protocol Signature Mismatch on Size
**What goes wrong:** CONTEXT.md specifies 256x256, 512x512, 1024x576 — none of which are supported by gpt-image-1 or Imagen 4
**Why it happens:** Protocol was designed around ideal game asset sizes; APIs have fixed size options
**How to avoid:** The `generate_image(width, height)` signature stays (for the GDScript reference intent), but each provider implementation maps internally to supported sizes. Document this in provider docstrings.
**Warning signs:** Returned images are the wrong aspect ratio for intended use

### Pitfall 7: `load_dotenv()` Not Finding Repo Root `.env`
**What goes wrong:** `OPENAI_API_KEY` is `None` even though `.env` exists; `AsyncOpenAI` raises `AuthenticationError`
**Why it happens:** `load_dotenv()` with no argument searches current working directory, not repo root
**How to avoid:** Always pass explicit path: `load_dotenv(Path(__file__).parent.parent.parent / ".env")` — three levels up from `backend/backend/main.py` reaches repo root
**Warning signs:** `openai.AuthenticationError: No API key provided` at pipeline start

## Open Questions

1. **Size mismatch between CONTEXT.md spec and API capabilities**
   - What we know: CONTEXT.md specifies 256x256 sprites, 1024x576 backgrounds, 512x512 textures. Neither gpt-image-1 nor Imagen 4 supports these exact sizes.
   - What's unclear: Whether GDScript scaling (via `Sprite2D.scale`) to 256x256 from a 1024x1024 image will produce acceptable quality pixel art.
   - Recommendation: All providers generate `1024x1024` and let GDScript scale down. Update Code Generator system prompt to specify `scale = Vector2(0.25, 0.25)` for sprites. Background landscape images use `1536x1024` mapped to `1024x576` viewport via texture rect or camera.

2. **Imagen 4 billing requirement for Google provider**
   - What we know: Imagen API requires a billing-enabled account; free tier does not cover image generation.
   - What's unclear: Whether a simple `GOOGLE_API_KEY` from AI Studio works or requires Vertex AI credentials.
   - Recommendation: Implement Google provider using `GOOGLE_API_KEY` with `genai.Client()`. Document billing requirement in `.env.example`. Default provider is `openai` which has a more accessible free tier. Test Google provider only with confirmed billing.

3. **Number of images per game for cost/latency balance**
   - What we know: 3-6 images per game (CONTEXT.md). gpt-image-1 at ~3-5 seconds per image; parallel = ~5-7 seconds total for 4 images.
   - What's unclear: Optimal count. More images = better game visuals but more cost and latency.
   - Recommendation: Default to 3 images per game: 1 character sprite, 1 background, 1 terrain/object texture. This stays within the 90-second budget easily and covers the main visual asset types.
