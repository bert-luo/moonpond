### Pattern 1: Base Method Name Set (Bug A)
**What:** Extract base method names from `_TEMPLATE_BASE` as a module-level constant, then filter `contract.game_manager_methods` against that set before iterating.
**When to use:** Any time contract methods are iterated in `_assemble_script()`.

```python
# Source: game_manager_generator.py (current _TEMPLATE_BASE analysis)

# Extract base function names from the template — these must never be duplicated
_TEMPLATE_BASE_METHOD_NAMES: frozenset[str] = frozenset({
    "_ready",
    "set_palette",
    "get_palette_color",
    "set_state",
})

def _extract_method_name(signature: str) -> str:
    """Extract bare function name from a GDScript method signature.

    e.g. "set_state(new_state: int) -> void" -> "set_state"
    e.g. "add_score(amount: int)" -> "add_score"
    """
    return signature.split("(")[0].strip()

# In _assemble_script(), before the "--- Methods ---" section:
contract_methods = [
    m for m in contract.game_manager_methods
    if _extract_method_name(m) not in _TEMPLATE_BASE_METHOD_NAMES
]
```

### Pattern 2: Autoload Deduplication Set (Bug B)
**What:** Build a set of hardcoded autoload names, then skip any contract autoloads already in that set.
**When to use:** In `_patch_project_godot_autoloads()` before emitting `autoload_lines`.

```python
# Source: wiring_generator.py (current _patch_project_godot_autoloads analysis)

_HARDCODED_AUTOLOAD_NAMES: frozenset[str] = frozenset({"GameManager"})

def _patch_project_godot_autoloads(template_content: str, autoloads: list[str]) -> str:
    autoload_lines = ["[autoload]\n", "\n", 'GameManager="*res://game_manager.gd"\n']
    for name in autoloads:
        if name in _HARDCODED_AUTOLOAD_NAMES:
            continue  # already emitted above
        script_snake = re.sub(r"(?<!^)(?=[A-Z])", "_", name).lower()
        autoload_lines.append(f'{name}="*res://{script_snake}.gd"\n')
    # ... rest unchanged
```

### Pattern 3: .tscn ExtResource Validation/Fixup (Bugs C + F)
**What:** After node files are generated, parse each .tscn file to verify every `ExtResource("X")` reference has a matching `[ext_resource ... id="X"]` declaration. Remove or fix orphan references. Optionally strip per-node .tscn files entirely (centralizing assembly in wiring stage).
**When to use:** Hookup point is `pipeline.py` after `run_parallel_node_generation()` returns.

Two valid approaches:

**Approach A — Fixup pass (keep per-node .tscn):**
```python
# New utility: backend/stages/tscn_validator.py

import re

def _extract_declared_ids(tscn: str) -> set[str]:
    return set(re.findall(r'\[ext_resource[^\]]*\bid="([^"]+)"', tscn))

def _extract_referenced_ids(tscn: str) -> set[str]:
    return set(re.findall(r'ExtResource\("([^"]+)"\)', tscn))

def validate_tscn_ext_resources(tscn: str) -> list[str]:
    """Return list of referenced IDs that have no matching declaration."""
    declared = _extract_declared_ids(tscn)
    referenced = _extract_referenced_ids(tscn)
    return sorted(referenced - declared)

def strip_tscn_files(node_files: dict[str, str]) -> dict[str, str]:
    """Remove all .tscn files from node output — let wiring stage own scene assembly."""
    return {k: v for k, v in node_files.items() if not k.endswith(".tscn")}
```

**Approach B — Centralize scene assembly in wiring stage (recommended for FIX-F too):**
Strip .tscn files from node output before passing to wiring generator. The wiring stage's LLM call already assembles `Main.tscn` from the full contract, so per-node .tscn files are redundant and error-prone. This eliminates both Bug C and Bug F in one pass.

The wiring generator's existing system prompt already instructs the LLM to include all nodes from the contract — it does not need per-node .tscn to function.

**Recommendation:** Approach B (strip per-node .tscn) is simpler, eliminates two bugs at once, and reduces LLM generation surface. The wiring stage already owns scene tree assembly.

```python
# In pipeline.py, after run_parallel_node_generation():
# Filter out per-node .tscn files — wiring stage owns all scene assembly
node_files = {k: v for k, v in node_files.items() if not k.endswith(".tscn")}
```

### Pattern 4: Signal Signatures in Prompt (Bug D)
**What:** The current `NodeContract.signals` field is `list[str]` storing bare signal names (e.g. `"score_changed"`). To fix argument mismatches, the prompt must include the full signal signature with arguments.

**Option A — Keep contract model unchanged, derive from existing data:**
The contract_generator LLM currently generates bare signal names. The system prompt for `contract_generator.py` could be updated to emit signal names WITH argument lists (e.g. `"score_changed(points: int)"`) and the node generator prompt already emits `f"Emit EXACTLY these signals: {node.signals}"`.

If signal names in the contract already carry the argument list string, the prompt is automatically correct with no structural changes.

**Option B — Add `signal_signatures: dict[str, str]` to NodeContract:**
Maps bare signal name to its argument list. More structured, but adds schema complexity.

**Recommendation:** Option A is simpler — update `contract_generator.py` system prompt to emit full signal signatures with arguments (e.g. `"score_changed(points: int)"` instead of `"score_changed"`). The `NodeContract.signals: list[str]` field can carry this enriched format without a schema change because it is a string list.

The existing node generator prompt already says `f"Emit EXACTLY these signals: {node.signals}"` — if those strings now include argument lists, signal emit calls will be correct.

### Pattern 5: NodeContract spawn_mode Field (Bug E)
**What:** Add a `spawn_mode` field to `NodeContract` distinguishing nodes that exist as static scene tree nodes from those that are dynamically spawned at runtime (via `instantiate()` / `add_child()`).
**When to use:** Wiring generator reads `spawn_mode` and skips dynamic nodes from `Main.tscn` scene tree entries.

```python
# In contract_models.py

from typing import Literal

class NodeContract(BaseModel):
    script_path: str
    scene_path: str | None = None
    node_type: str
    description: str
    methods: list[str] = []
    signals: list[str] = []
    groups: list[str] = []
    dependencies: list[str] = []
    spawn_mode: Literal["static", "dynamic"] = "static"  # NEW FIELD
```

The contract generator system prompt must be updated to set `spawn_mode: "dynamic"` for nodes described as dynamically instantiated (e.g. bullets, enemies, particles). The wiring generator must filter:
```python
static_nodes = [n for n in contract.nodes if n.spawn_mode == "static"]
```
and pass only `static_nodes` in the `[node]` entries of `Main.tscn`.

### Anti-Patterns to Avoid
- **Parsing .tscn with a full tokenizer:** The Godot text scene format is line-oriented and regex-parseable for the specific patterns here. Don't implement a full parser — targeted regex patterns are sufficient and already match the pattern used throughout the codebase.
- **LLM for .tscn fixup:** Do not feed broken .tscn back to the LLM for repair. The fixup is deterministic (strip per-node .tscn, let wiring stage own assembly).
- **Removing base method names by string prefix:** Use exact set membership, not substring matching, to avoid accidentally filtering `set_state_from_save()` when `set_state` is the forbidden name.
- **signal_signatures as a new model field when the string list suffices:** Adding `dict[str, str]` complicates the schema needlessly. Enriched strings in `list[str]` are sufficient.

## Common Pitfalls

### Pitfall 1: Method Name Extraction from Signatures
**What goes wrong:** `contract.game_manager_methods` contains full GDScript signatures like `"set_state(new_state: int) -> void"`. A naive string comparison `"set_state" in sig` would match `"set_state_from_network(x)"`.
**Why it happens:** Signal/method signatures are enriched strings, not bare names.
**How to avoid:** Use `sig.split("(")[0].strip()` to extract the bare name before comparing against the forbidden set.
**Warning signs:** Test `test_gamestate_enum_from_contract` already passes — check that a new test with `"set_state_extended()"` is also correctly allowed.

### Pitfall 2: .tscn String ID vs Integer ID
**What goes wrong:** Godot 4 text format uses string IDs in quotes: `id="1"`, not `id=1`. Regex patterns must match the quoted form.
**Why it happens:** Existing test `test_ext_resource_ids_unique` already uses `r'\[ext_resource[^\]]*id="(\d+)"'` — this is correct. Per-node .tscn files generated by the LLM may use the same pattern or may invent non-integer IDs like `id="player_script"`.
**How to avoid:** Match `id="([^"]+)"` (any string, not just digits) to handle both integer and string IDs.
**Warning signs:** If the fixup pass incorrectly reports all IDs as mismatched, the regex is likely matching the wrong quote pattern.

### Pitfall 3: Contract Generator Not Updated for FIX-D
**What goes wrong:** Even if the node generator prompt correctly says "emit signals with these exact signatures", the contract generator LLM must produce signal strings that include argument types. If `contract_generator.py`'s prompt still asks for bare signal names, the signals list will still be bare.
**Why it happens:** FIX-D requires coordinated changes at TWO LLM call sites: the contract generator (which produces signal names) and the node generator (which uses them in emit calls).
**How to avoid:** Update `contract_generator.py` system prompt to require full signal signatures with argument types (e.g. `"bird_flapped(velocity: Vector2)"` not `"bird_flapped"`).

### Pitfall 4: spawn_mode Default Breaks Existing Tests
**What goes wrong:** Adding `spawn_mode: Literal["static", "dynamic"] = "static"` to NodeContract is backward-compatible (default "static"), but wiring generator tests that construct contracts without `spawn_mode` will now silently skip filtering. This is correct behavior for existing tests but should be verified.
**Why it happens:** Pydantic model with a default value is backward-compatible.
**How to avoid:** Existing tests pass contracts with all `spawn_mode="static"` implicitly — no changes needed to existing tests. New tests for FIX-E should create contracts with `spawn_mode="dynamic"` nodes and verify they are excluded from Main.tscn.

### Pitfall 5: Stripping .tscn May Remove Files Wiring Stage Needs
**What goes wrong:** If the wiring stage's LLM prompt references specific sub-scene paths from per-node .tscn files, stripping them may cause broken references.
**Why it happens:** The wiring stage system prompt currently passes `generated_files.keys()` (file names) to the LLM. If per-node .tscn paths were referenced there, the LLM might have used them.
**How to avoid:** The current `_build_wiring_system_prompt()` passes `json.dumps(sorted(generated_files.keys()))` — verify that stripping .tscn keys does not break the wiring LLM's ability to generate Main.tscn. The wiring LLM uses the full `GameContract` (not the file list) to determine scene structure, so this should be safe.
