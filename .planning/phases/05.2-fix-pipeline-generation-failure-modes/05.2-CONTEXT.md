# Phase 05.2: Fix Pipeline Generation Failure Modes - Context

**Gathered:** 2026-03-18
**Status:** Ready for planning
**Source:** Failure analysis from flapventure-wings-of-destiny game output

<domain>
## Phase Boundary

This phase fixes the remaining pipeline generation bugs that cause blank screens, runtime crashes, and logic errors in generated games. All bugs were identified from inspecting a real pipeline output (Flapventure: Wings of Destiny, 2026-03-17). Phase 05.1 already fixed context enrichment — this phase addresses the failure modes that enrichment alone doesn't solve.

</domain>

<decisions>
## Implementation Decisions

### Bug A: Duplicate method definitions in game_manager.gd (FATAL)
- The game_manager_generator must filter out any contract methods whose names collide with template base methods (`set_state`, `set_palette`, `get_palette_color`, `_ready`)
- Extract base method names as a constant and filter before generating method bodies
- Location: `game_manager_generator.py` `_assemble_script()`

### Bug B: Duplicate GameManager autoload in project.godot
- `_patch_project_godot_autoloads()` in wiring_generator must deduplicate — build a set of autoload names before emitting lines
- Skip contract autoloads already in the hardcoded list
- Location: `wiring_generator.py`

### Bug C: Invalid ExtResource IDs in node .tscn files (FATAL)
- Add a .tscn validation/fixup pass after node generation
- Every `ExtResource("X")` reference must have a matching `[ext_resource ... id="X"]` declaration
- Consider centralizing all scene assembly in the wiring stage rather than trusting per-node LLM .tscn output

### Bug D: Signal argument mismatches
- Include signal signatures (with parameter names and types) in the method generation prompt
- The LLM must know to emit `signal_name.emit(arg1, arg2)` with correct arguments
- Add signal parameter definitions to the contract model if not already present
- Location: node_generator.py prompt construction

### Bug E: Dynamic vs static node collision (PipePairContainer)
- Define ownership rules: if a script creates a node dynamically, the wiring stage must not also create it in Main.tscn
- Contract-level: add a field to NodeContract distinguishing static scene nodes from dynamically-spawned nodes
- Location: contract model + wiring_generator.py

### Bug F: Script attached to wrong node in .tscn
- Same fix path as Bug C — .tscn validation or centralized scene assembly
- Validate that scripts are only attached to root nodes in generated .tscn files

### Claude's Discretion
- Whether to add a general .tscn validator utility vs inline validation in generators
- Whether to centralize ALL scene assembly in wiring stage (removing per-node .tscn generation) or keep per-node .tscn with validation
- Test strategy for each fix (unit tests on generators vs integration tests on pipeline output)
- Whether signal signatures need a contract model change or can be derived from existing contract data

</decisions>

<specifics>
## Specific Ideas

### Plan grouping from failure analysis
- **Plan 1 (deterministic fixes):** Bugs A + B — pure code fixes in generators, no LLM prompt changes
- **Plan 2 (prompt/context fixes):** Bug D — enrich method generation prompt with signal signatures
- **Plan 3 (.tscn reliability):** Bugs C + F — validate/fixup .tscn output or centralize scene assembly in wiring stage
- **Plan 4 (contract coordination):** Bug E — define ownership rules for dynamic vs static nodes

### Priority ordering

| C: Invalid ExtResource | Fatal | Medium |
| D: Signal arg mismatch | Runtime | Low |
| E: Container collision | Logic | Medium |
| F: Script on wrong node | Runtime | Medium |
