<task type="auto">
  <name>Task 1: Implement Code Generator stage</name>
  <files>backend/backend/stages/code_generator.py</files>
  <action>
Create `backend/backend/stages/code_generator.py`:

- Define `SONNET_MODEL = "claude-sonnet-4-6"`
- Define a comprehensive system prompt (`_CODEGEN_SYSTEM_PROMPT`) that instructs the LLM to:
  - Generate GDScript files for a Godot 4 game based on a GameDesign specification
  - Use ONLY Godot 4 GDScript syntax (NOT GDScript 3 or Python)
  - Explicitly list forbidden patterns: `True/False` (use `true/false`), `None` (use `null`), `self.variable` for member vars (use just `variable`), `Input.is_key_pressed(KEY_X)` (use `Input.is_action_pressed("action_name")`)
  - List the 8 valid input action names: move_left, move_right, move_up, move_down, jump, shoot, interact, pause
  - Respond ONLY with a JSON object where keys are filenames (e.g., "main.gd", "player.gd") and values are the full GDScript source code
  - Do NOT include markdown code fences — respond with raw JSON only
  - Each script should extend an appropriate Godot node type (Node2D, CharacterBody2D, Area2D, etc.)

- Define a helper `_build_codegen_prompt(game_design: GameDesign, previous_error: str | None = None) -> str` that:
  - Serializes the GameDesign as JSON
  - If `game_design.control_scheme` is NOT WASD, reads the relevant control snippet source from disk (using CONTROL_SNIPPET_PATHS) and includes it in the prompt with instructions to attach/integrate it
  - If `previous_error` is provided, prepend: "PREVIOUS ATTEMPT FAILED. Fix these errors:\n{previous_error}\n\nGenerate corrected code:"

- Implement `async def run_code_generator(client: AsyncAnthropic, game_design: GameDesign, emit: EmitFn, previous_error: str | None = None) -> dict[str, str]`:
  - Only emit `ProgressEvent(type="stage_start", message="Writing game code...")` if `previous_error is None` (avoid duplicate stage_start on retries per Pitfall 4 from research)
  - Call `await client.messages.create()` with model=SONNET_MODEL, max_tokens=8192, system prompt, user message from `_build_codegen_prompt`
  - Parse response text as JSON (strip markdown code fences if present)
  - Return `dict[str, str]` of filename -> GDScript content

- Define `_check_gdscript_syntax_patterns(files: dict[str, str]) -> str | None` that checks all generated GDScript content for common Python contamination:
  - `True` or `False` (not `true`/`false`) — but only as standalone tokens, not inside strings
  - `None` (not `null`) — same caveat
  - `Input.is_key_pressed` (should use `Input.is_action_pressed`)
  - Returns a human-readable error string if issues found, None if clean

NOTE on control snippet injection: For non-WASD control schemes, read the control snippet source file from the template directory on disk. The path is relative to repo root: `godot/templates/base_2d/assets/control_snippets/{name}.gd`. Read the file content and include it in the prompt so the LLM can reference the actual GDScript implementation.

1. Create `backend/backend/stages/visual_polisher.py`:

- Define `SONNET_MODEL = "claude-sonnet-4-6"`
- Define a system prompt (`_POLISHER_SYSTEM_PROMPT`) that instructs the LLM to:
  - Review GDScript files and add visual polish using the template asset library
  - MUST add at least one shader reference and one palette selection (per STAGE-04 requirement)
  - MAY add particle scene references where appropriate
  - Include the EXACT `res://` paths for all available assets (copy from models.py constants):
    - Shaders: res://assets/shaders/pixel_art.gdshader, etc.
    - Palettes: res://assets/palettes/neon.tres, etc.
    - Particles: res://assets/particles/explosion.tscn, etc.
  - Show how to apply a shader: `var shader = preload("res://assets/shaders/pixel_art.gdshader")` then set on a ShaderMaterial
  - Show how to apply a palette: `var palette = preload("res://assets/palettes/neon.tres")` — gradient resource
  - Show how to instance a particle: `var particles = preload("res://assets/particles/dust.tscn").instantiate()`
  - Respond with a JSON object: keys = filenames, values = the COMPLETE patched GDScript (not diffs)
  - Do NOT modify the control logic or game mechanics — only add visual enhancements
  - Never write to `res://assets/` — only modify script files

- Implement `async def run_visual_polisher(client: AsyncAnthropic, files: dict[str, str], visual_style: VisualStyle, emit: EmitFn) -> dict[str, str]`:
  - Emit `ProgressEvent(type="stage_start", message="Adding visual polish...")`
  - Build user message containing: the current files dict as JSON, and the VisualStyle (palette, shader, mood) to guide choices
  - Call `await client.messages.create()` with model=SONNET_MODEL, max_tokens=8192
  - Parse response as JSON dict
  - Return the patched files dict

2. Create `backend/backend/stages/exporter.py`:

- Import `shutil`, `Path`, `run_headless_export` from `...godot.runner`, `ProgressEvent`/`GameResult` from `...pipelines.base`, `EmitFn`
- Define path constants:
  ```python
  _REPO_ROOT = Path(__file__).parent.parent.parent.parent  # backend/backend/stages -> repo root
  TEMPLATE_DIR = _REPO_ROOT / "godot" / "templates" / "base_2d"
  GAMES_DIR = _REPO_ROOT / "games"
  ```
  NOTE: The path traversal from `stages/exporter.py` needs 4 `.parent` calls: stages -> backend -> backend -> repo_root. Verify this matches the actual nesting depth.

- Implement `async def run_exporter(job_id: str, files: dict[str, str], controls: list[dict], emit: EmitFn) -> GameResult`:
  - Emit `ProgressEvent(type="stage_start", message="Building for web...")`
  - Set `project_dir = GAMES_DIR / job_id / "project"` and `export_dir = GAMES_DIR / job_id / "export"`
  - Copy template: `shutil.copytree(TEMPLATE_DIR, project_dir, dirs_exist_ok=True)` (use dirs_exist_ok=True per Pitfall 6 from research)
  - Create `scripts_dir = project_dir / "scripts"`, `scripts_dir.mkdir(exist_ok=True)`
  - Write each file from the `files` dict into `scripts_dir / filename`
  - Call `result = await run_headless_export(project_dir, export_dir)`
  - If `not result.success`: raise `RuntimeError(f"Export failed: {result.stderr[:500]}")`
  - Return `GameResult(job_id=job_id, wasm_path=f"/games/{job_id}/export/index.html", controls=controls)`

NOTE: Exporter does NOT call any LLM. It uses the existing Phase 2 `run_headless_export()` runner. The `controls` parameter is a list of dicts (from GameDesign.controls mapped to dicts) passed through to GameResult for the frontend.
